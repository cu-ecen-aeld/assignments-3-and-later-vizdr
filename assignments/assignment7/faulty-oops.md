## Information from the oops

The first line indicates the nature of the oops, an attempt to dereference a NULL pointer:
> `Unable to handle kernel NULL pointer dereference at virtual address 0000000000000000`The instruction that caused the oops is reported as the value of the pc register at the time of the oops:

The instruction that caused the oops is reported as the value of the pc register at the time of the oops:
pc : faulty_write+0x10/0x20 [faulty]

The oops happened when running the function faulty_write, in module [faulty], 0x10 bytes from the start of the function in the binary.

To find out what the instruction was, the `faulty.ko` file is to be disassembled using `objdump`. In this case, the target did not have objdump, so the following command should run on the host machine using the objdump from the appropriate cross compiler toolchain.

$ aarch64-none-linux-gnu-objdump -d buildroot/output/build/ldd-...
